        Why it's "spec-driven"
        
3 กลไกที่ทำให้ flow นี้ทำงาน

        
ไม่ใช่แค่ลำดับคำสั่ง — แต่คือกติกาที่ผูกทุกสเตจเข้าหากันและกันเอาไว้ผ่าน artifact

      

      

        

          
01 · Binds everything

          
Constitution เหนือทุกสเตจ

          
กติกาที่ตั้งใน /constitution ถูกบังคับตอน /plan (gate) และแปลงเป็นลำดับงานใน /tasks — เป็นสิ่งที่ Thoughtworks ชี้ว่าเป็นเอกลักษณ์ของ Spec Kit

        

        

          
02 · Test before code

          
TDD ฝังในลำดับงาน

          
เทสต์ต้องแดงก่อน (Phase B) แล้วโค้ดจริงถึงตามมา (Phase C) — บังคับโดยหลัก "Test-First" ในรัฐธรรมนูญ ไม่ใช่แค่คำแนะนำ

        

        

          
03 · Human as validator

          
คนคุมที่จุดต่อ

          
มนุษย์ตัดสินใจที่ /clarify (ตอบคำถาม) และ review ระหว่าง /implement — AI เป็น engine, คนเป็นผู้ตรวจสอบ

        

      

    

  


  
  

    

      

        The thread
        
ตามรอย 1 requirement ทะลุทุก artifact

        
จุดแข็งของ spec-driven คือ "ความสืบสาวได้" — ดูว่ากฎ "ชื่อ ≤200 ตัวอักษร" เดินทางจาก spec ไปจนถึงเทสต์ยังไง

      


      

        

          

            
spec.md

            
FR-009 ชื่อ task ต้องยาว ≤ 200 ตัวอักษร

          

          
→

          

            
clarify

            
Q1 → ตอบ "200" (เคลียร์ NEEDS CLARIFICATION)

          

          
→

          

            
data-model.md

            
title CHECK(length 1–200)

          

          
→

          

            
openapi.yaml

            
title maxLength: 200

          

          
→

          

            
tasks.md

            
T007 unit test: ยาว >200 → 422

          

        

      

      
= ทุกบรรทัดโค้ดสาวกลับไปหา requirement ได้ — ไม่มีอะไร "โผล่มาเฉย ๆ" ในโค้ด ทุกอย่างมีที่มาจาก spec

    

  


  
  

    

      

        จำไว้
        
Spec Kit ไม่ได้ทำให้คนหายไป — แต่ย้ายบทบาทคนไปอยู่ที่ "จุดตัดสินใจ" แทนการนั่งพิมพ์โค้ดทุกบรรทัด: เขียนกติกา (constitution), ตอบคำถามที่กำกวม (clarify), และ review PR (implement) นี่คือหลัก human-as-validator ที่เป็นหัวใจของทุก AI-SDLC framework
